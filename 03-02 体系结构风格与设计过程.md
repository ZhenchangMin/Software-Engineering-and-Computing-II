# 03-02 体系结构风格与设计过程

> 软件工程与计算 II · 体系结构模块 · 第 2 讲
> **核心三问**：① 常见体系结构风格有哪些，各自适用场景？ ② 分层风格和 MVC 如何指导设计？ ③ 从需求到架构的设计过程包含哪些步骤？

## PART 1：体系结构风格概览

> 体系结构风格 = **组件词汇 + 连接器词汇 + 拓扑约束**。它规定一组组件类型、组件连接方式、交互的约束规则。
> 为什么需要风格：提供经验复用、通用词汇、明确质量属性权衡。

### 风格分类总览

| 风格 | 组件 | 连接器 | 典型应用 |
|------|------|------|------|
| 分层 Layered | 层 | 过程调用 | 企业信息系统 |
| 管道-过滤器 | 过滤器 | 管道（数据流） | 编译器、数据处理 |
| 面向对象 | 对象/类 | 方法调用、消息 | 现代软件、框架 |
| 客户端-服务器 | 客户端、服务器 | 请求/响应 | Web、数据库 |
| 事件驱动 | 事件源、监听器 | 事件总线 | GUI、消息系统 |
| MVC | Model/View/Controller | 事件、直接调用 | Web、桌面 GUI |
| 微服务 | 独立服务 | REST/消息队列 | 大规模互联网 |
| 主程序-子程序 | 主程序、子程序 | 过程调用 | 传统结构化系统 |

### 风格 vs 模式

| 维度 | 体系结构风格 | 设计模式 |
|------|------|------|
| 作用范围 | 系统全局组织方式 | 局部重复问题的解决方案 |
| 粒度 | 定义系统大部分的组织 | 应用于单个元素 |
| 能否共存 | 通常选一种主风格 | 多个模式可同时存在 |

> 案例：在线书店 = 三层分层（全局）+ MVC（展示层）+ Observer（模型通知视图）+ DAO（数据访问）。

### KWIC 实验（Parnas 1972）

用 5 种风格实现同一问题（上下文关键词索引），对比可修改性/可复用性/性能：

| 风格 | 数据共享 | 耦合 | 核心特征 |
|------|------|------|------|
| MS（主程序-子程序）| 共享存储 | 紧 | 空间高效，数据无隐藏 |
| OO（面向对象）| 接口访问、内部隐藏 | 松 | 信息隐藏，易变更数据表示 |
| PF（管道-过滤器）| 管道传输、无共享 | 极松 | 增量处理，但仅限批处理 |
| ES（事件系统）| 抽象数据空间 | 松 | 灵活扩展，但控制流难追踪 |

> **核心启示：没有最好的风格，只有最适合需求的风格。**

## PART 2：分层风格

- **严格分层**：每层只调用直接下层（如 OSI 七层）。
- **松散分层**：允许跨层调用，提高效率（大多数企业应用）。

### 三层架构

| 层次 | 核心职责 | 典型技术 |
|------|------|------|
| 展示层 Presentation | 接收请求、渲染视图、用户交互 | Vue, React, Thymeleaf |
| 业务逻辑层 Business Logic | 实现业务规则、协调工作流 | Spring Service |
| 数据访问层 Data Access | CRUD、数据映射、事务 | MyBatis, JPA |

> **关键约束**：上层依赖下层、下层不依赖上层；展示层不直接访问数据库；各层通过**接口**解耦。

**四大优势**：封装性（切换 MySQL→PostgreSQL 不影响业务层）、关注点分离、可替换性、可测试性。
**代价**：性能开销、级联修改、过度分层（简单系统反增复杂度）。

```java
// 展示层 → 业务层 → 数据层
@RestController class BookController { @Autowired BookService s;
  @GetMapping("/books/{id}") Book get(@PathVariable Long id){ return s.findById(id); } }
@Service class BookService { @Autowired BookDao d;
  Book findById(Long id){ return d.selectById(id); } }
@Mapper interface BookDao { @Select("...") Book selectById(Long id); }
```

## PART 3：MVC 风格

| 组件 | 职责 | 要点 |
|------|------|------|
| Model | 管理数据与业务逻辑 | 不依赖 View/Controller |
| View | 展示数据 | 从 Model 获取数据 |
| Controller | 处理用户输入 | 更新 Model，选择 View |

> **MVC 与三层的映射**：View+Controller → 展示层；Model → 业务逻辑层+数据访问层。分层关注纵向职责划分，MVC 关注交互逻辑分离，两者可结合。

> **观察者模式在 MVC 中**：Model 是被观察者(Subject)，View 是观察者(Observer)。Model 数据变化时通知所有 View 自动刷新——支持多视图同步，新增 View 无需改 Model。

**其他风格补充**：管道-过滤器（Java Stream API、Unix 管道）；C/S + REST（无状态、统一接口、分层系统）；微服务（独立部署/独立数据库，5 人团队+无 K8s 经验→选单体分层更合适）。

## PART 4：体系结构设计过程

> 设计过程是**迭代的**，任何步骤都可回退调整。

| 步骤 | 名称 | 核心任务 |
|------|------|------|
| 1 | 分析关键需求 | 识别功能需求、质量属性、项目约束 |
| 2 | 选择体系结构风格 | 根据需求特征选择 |
| 3 | **逻辑设计** | 将需求分配到子系统和模块（最核心，反复迭代）|
| 4 | 接口设计 | 定义模块间交互接口 |
| 5 | 物理设计 | 决定部署方案 |
| 6 | 验证设计方案 | 检查是否满足需求和约束 |
| 7 | 评审与改进 | 团队评审、迭代优化 |

### ASR——架构重要需求

> Architecturally Significant Requirement：影响软件体系结构的需求。**并非所有需求都是 ASR**，只有驱动架构决策的才是。
> 例：「支持 1000 并发」「支付加密」「7×24 可用」「必须用 Java」是 ASR；「按书名搜索」「按钮颜色蓝色」非 ASR。
> 非功能需求和项目约束对架构选择的影响往往大于功能需求。

### 架构设计三阶段循环（Hofmeister 2007）

> **Analysis（收集 ASR）→ Synthesis（构建候选方案、权衡）→ Evaluation（验证是否满足 ASR）**，迭代进行。是所有架构方法（ADD/ATAM/QAW）的底层模型。

经典方法：**QAW**（质量属性工作坊，"设计目标是什么"）→ **ADD**（属性驱动设计，"如何根据目标做设计"）→ **ATAM**（架构权衡分析，"设计方案是否可行"，输出风险点/敏感点/权衡点）。

### 逻辑设计原则

> 高内聚、低耦合、单一职责、信息隐藏。步骤：识别功能域 → 为每个功能域创建三层模块 → 定义依赖关系 → 检查模块粒度。

## PART 5：案例——连锁超市从需求到架构

需求分析：10 个功能需求 + 非功能需求（安全：认证/权限/审计；约束：兼容 ERP）+ 项目约束（Java+Spring、3 个月、5 人）。

> **选择分层风格的理由**：企业级业务系统、Java+Spring 天然分层、安全可作独立层、功能模块多需清晰职责、进度紧分层成熟度高。决策：三层分层 + 松散分层。

设计方案演进：
- **方案一（每层一个包）**：数据层粒度太粗，违反单一职责，团队并行开发困难。
- **方案二（按功能域拆包）**：三层均按功能域拆为独立子包，每层约 5 个包，支持并行开发。
- **方案三（接口独立层 + 依赖倒置）**：Service/DAO 接口独立成层，实现继承接口，上层依赖接口；VO（展示↔逻辑）与 PO（逻辑↔数据）分离。

## PART 6：AI 增强——架构风格选择

> LLM × 软件架构研究：主要应用于初期阶段（Requirements-to-Architecture、Architecture-to-Code），Few-shot + RAG 是主要技术，通常优于基线但复杂架构推理仍有局限。

工业实践：AWS Kiro（Spec→Design→Tasks→Code）、GitHub Spec Kit、Claude Code（CLAUDE.md 作"宪法"+ Subagent 并行）。

> **推荐做法**：让 Agent 生成架构方案 → 人工 Review → 记录 ADR → 再让 Agent 实现。

AI 对比分析（分层 vs 微服务）结论：5 人团队、3 个月、Java → **分层架构最优**，微服务引入的复杂度远超收益。

**人工审查 AI 架构建议检查清单**：需求覆盖、约束一致、经验验证、风险评估、替代方案、演化能力、技术生态。

> 切记：AI 可能缺乏项目上下文，给出看似合理但不适合具体场景的建议。**AI 生成的架构建议必须经过人工审查。**

## 课堂总结

1. 选择架构风格要基于**需求特征和项目约束**，而非技术偏好。
2. 逻辑设计的关键是合理的**模块粒度和清晰的职责划分**。
3. 架构设计是**迭代过程**，不是一次性完成。
