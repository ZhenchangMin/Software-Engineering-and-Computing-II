# 01-01 软件工程基础

> 软件工程与计算 II · 模块一 · 单元 1
> **核心三问**：① 软件与程序有何区别？软件三要素是什么？ ② 软件工程的定义是什么、为何称为"工程"？ ③ SWEBOK V4 新增了哪些方向？

## PART 1：从一次 5 亿美元的失败开始

### Ariane 5 · 37 秒后爆炸

| 项 | 内容 |
|----|------|
| 时间 | 1996-06-04，阿丽亚娜 5 型火箭首飞，发射后 37 秒解体自毁 |
| 调查结论 | 64 位浮点数转换为 16 位整数时溢出，惯性参考系统（SRI）崩溃 |
| 根本原因 | 直接复用 Ariane 4 的 SRI 模块，未在新环境重新验证 |
| 代价 | 5 亿美元火箭与卫星载荷 + 数年研发延期 |

> **"代码完全正确，工程完全失败"的典型案例。** SRI 模块通过了所有测试——但没有人验证它是否适合 Ariane 5 的新飞行轨迹。

### 灾难的传导链

```
非技术约束（进度与成本压力）
  "Ariane 4 的 SRI 已验证，直接复用节省时间"
        ↓ 隐性假设 > 工程验证
  数值范围假设未更新      ← 隐式约束缺失
  集成测试不含新飞行场景  ← 验证遗漏
  单点故障无容错设计      ← 系统约束缺失
        ↓ 软件"忠实"执行了错误的假设
  SRI 溢出宕机 → 控制系统接收无效数据 → 触发自毁指令
        ↓ 37 秒
  首飞失败 = 5 亿美元 + 4 颗卫星
```

> 需求工程术语：此处溢出源于**隐式约束（Implicit Constraint）**未被显式记录。根因不是"程序员写错代码"，而是整个软件工程过程在关键环节系统性失效。

## PART 2：软件是什么？

### 一条变革之路

```
Information ── Computing ── Digital Computer ── Software ── AI ?
```

软件的本质是**信息处理**——从泥板到云端，变的是介质（记录/交流/存储），不变的是对信息的处理。

软件认知的历史演变：

| 年代 | 认知 | 标志 |
|------|------|------|
| 1930s | 理论奠基 | 图灵《论可计算数》、香农布尔逻辑 |
| 1940s | 软件是硬件的一部分 | ENIAC、EDVAC（存储程序概念→冯·诺依曼架构）|
| 1950s | 软件工程 ≈ 硬件工程 | FORTRAN/LISP/COBOL 诞生；1958 Tukey 首创 "Software" 一词 |
| 1960s | 软件 ≠ 硬件 | 认识到软件与现实关系密切、易修改、无损耗、不可见 |
| 1970s~80s | Program = Algorithm + Data Structure | Niklaus Wirth：算法与数据结构天然关联 |
| 1990s~ | 软件开发远比编程复杂 | 万维网、移动电商爆发 |

### 编码只是冰山一角

软件工作量随规模变化（Jones 1996）：规模越大，编码占比越小。

| 规模(KLOC) | 编码% | 文档% | 缺陷清除% | 管理支持% |
|------|------|------|------|------|
| 0.1 | 70 | 5 | 15 | 10 |
| 100 | 30 | 26 | 30 | 14 |
| 1,000 | 18 | 31 | 35 | 16 |

> 100 万 LOC 大型系统成本排序（Jones 2007）：**1.缺陷清除 2.文档编写 3.会议交流 4.编码 5.项目管理 6.变更控制**——编码仅排第四！

### 软件 ≠ 程序

```
软件 = 程序(Program) + 数据(Data) + 文档(Document) + 知识(Knowledge)
```

| 要素 | 内容 | Ariane 5 缺失了什么 |
|------|------|------|
| 程序 | 可执行的指令序列 | SRI 代码本身正确 ✓ |
| 数据 | 程序处理与存储的信息 | 飞行参数范围未更新 ✗ |
| 文档 | 开发、使用、维护说明 | 数值范围约束未记入文档 ✗ |
| 知识 | 隐性经验、最佳实践、领域理解 | 领域知识未被显式化传承 ✗ |

> **IEEE 610.12-1990**：软件是计算机程序、规程及相关文档和运行所需数据的完整集合。
> 现代扩展定义额外加入"知识"要素。

### 软件与硬件：七大本质差异

| 特征 | 说明 | 工程含义 |
|------|------|------|
| 无形性 | 看不见摸不着 | 进度与质量难以直接观测 |
| 复杂性 | 逻辑交织 | 需要抽象、分层、模块化 |
| 可复制性 | 复制成本趋近于零 | 分发成本低，侵权风险高 |
| 易修改性 | 改动方便但影响难预测 | 变更管理至关重要 |
| 退化性 | 不磨损，但随环境"腐化" | 技术债 · 维护成本持续增长 |
| 依赖性 | 依赖运行环境(OS/库/网络) | 环境一致性是工程问题 |
| 智力密集 | 本质是人的智力产品 | 人才是最核心的生产要素 |

> Ariane 5 的**依赖性**体现：SRI 模块依赖"Ariane 4 飞行参数范围"这一隐性环境假设。

### 软件形态的六十年演变

```
1960s    1980s     2000s      2010s       2020s      2025+
大型机 → 桌面应用 → Web B/S → 移动互联网 → 云原生 → AI Agent
批处理   C/S架构   J2EE·SOA   App Store   微服务    自然语言即接口
```

**传统三种分类**：系统软件（Linux/驱动）、编程软件（VS Code/Git）、应用软件（微信/银行核心）。

**云原生四要素（CNCF）**：微服务 · 容器化 · DevOps · 持续交付。交付层次：IaaS（你管应用代码）→ PaaS（你管应用+数据）→ SaaS（你只管使用）。Serverless = FaaS + BaaS。

### LLM 时代：软件形态的第五次跃迁

四种交互模式演进：① 问答（ChatGPT）→ ② 问数（Text-to-SQL）→ ③ 助手（Copilot/Cursor）→ ④ 智能体（AutoGPT/Manus）。

| 新范式 | 含义 |
|------|------|
| Agent as Service | 服务单元从 API 接口变为"会思考的自主代理" |
| Context as Service | 记忆与上下文跨会话持久化，成为产品核心资产 |
| 抛弃式交互 | 生成即用、用完即弃，无需长期维护 |

> **应用软件基于现实又高于现实**：来源于现实（开发意图源于现实问题）→ 基于现实（必须基于现实才能解决问题）→ 改进现实（最终用于并改进现实）。

## PART 3：什么是软件工程？

### 软件危机与工程化回应

| 年代 | 标志性事件 |
|------|------|
| 1968 | NATO 软件工程会议，"软件危机"概念提出 |
| 1970s | 瀑布模型，引入工程化流程 |
| 1980s | 结构化方法、CMMI |
| 1990s | 面向对象、统一过程 RUP |
| 2000s | 敏捷宣言、Scrum/XP |
| 2010s | DevOps · 云原生 |
| 2020s | AI for SE · SE for AI |

> Barry Boehm（2006）：软件工程的每一次革命，都是对前一阶段未解决问题的系统性回应。

### 软件工程的定义（IEEE 610.12-1990）

> 将**系统化的、规范化的、可量化的**方法应用于软件的开发、运行和维护的过程；即将工程化的方法应用于软件。

### 工程学科的发展规律（M. Shaw, 1990）

```
经验积累 ──→ 技术实践 ──→ 理论指导 ──→ 成熟工程学科
铁匠铺式    作坊式      工业化      职业化
代码黑客    方法论      模型·理论    规范·认证
```

> 软件工程目前处于**理论指导 → 成熟工程**的过渡阶段。

**工程的四大要素**：Problem（动机）、Scientific Knowledge（工具/科学基础）、Solution/Machine（目标）、Cost-Benefit Effective（条件——追求足够好，而非最好）。

> **Engineering = Science + Principle + Art**。软件工程追求足够好，不是最好；真正的产品是基于虚拟计算机的软件方案；最终目的是促进整个社会进步。

### SWEBOK V4（2024）重大更新

核心 11 个知识域保留（需求/设计/构造/测试/维护/配置管理/工程管理/过程/质量等），**V4 新增 5 个方向**：

| 新增知识域 | 意义 |
|------|------|
| 软件架构 | 从设计中独立，地位提升 |
| 软件工程运维 | DevOps 时代运维纳入 SE |
| 软件安全 | 从附录升级为独立知识域 |
| AI for SE | AI 辅助软件工程全过程 |
| SE for AI | AI 系统的工程化方法 |

### 软件开发：活动 · 产物 · 角色

```
需求工程 ── 系统设计 ── 编码实现 ── 测试验证 ── 运维部署
 需求文档   架构文档    源代码     测试报告    运行日志
```

| 活动 | 目标 | 产物 |
|------|------|------|
| Requirement | What（做什么） | SRS 需求规格说明书 |
| Design | How（怎么做） | SDD 设计文档 |
| Construction | Build（构建） | 代码与可执行文件 |
| Testing | 做对的事 & 把事做对 | 测试报告 |
| Deliver | Install（部署） | 用户/系统文档 |
| Maintenance | Evolution（演化） | 新版本软件 |

核心角色：产品经理（定义做什么）、架构师（定义怎么做）、开发工程师、测试工程师、运维工程师。

> **LLM 时代的不变内核**：理解用户真实需求 · 管理系统复杂度 · 保障质量与可靠性 · 工程伦理——AI 改变的是效率与手段，对正确性与用户价值的责任只增不减。

## PART 4：今天最重要的三个结论

1. **软件 ≠ 程序**：文档与数据同样是软件核心组成；缺少文档的软件，是下一次 Ariane 5 事故的隐患。
2. **工程方法来自失败的积累**：每一条软件工程规范背后都有一次代价高昂的教训。
3. **AI 改变工具，不改变工程本质**：AI for SE 让工具更智能，但系统化·规范化·可量化的工程原则不变。

> "写出正确的代码只是第一步——让正确的代码在正确的地方运行，才是工程。"
