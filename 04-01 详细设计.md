# 04-01 详细设计：从体系结构到代码

> 软件工程与计算 II · 详细设计模块 · 第 1 讲
> **核心三问**：① 详细设计的输入/输出是什么？在 SDLC 中的位置？ ② 面向对象详细设计如何通过职责和协作建立静态/动态模型？ ③ 如何用 GRASP 模式指导职责分配？如何选择控制风格？

## PART 1：详细设计基础

> **详细设计 = 中层设计（过程、调用；类、协作）+ 低层设计（数据结构、算法、类型、语句、控制结构）**。
> 体系结构定义模块的规格（部件、连接件、配置），详细设计用详细设计机制实现模块。

**设计是一种问题求解**，软件设计尤其面对 **Wicked Problems**（无确定解，只有更好或更差的权衡）。五步设计过程（Crystallize→Formulate→Devise→Introduce→Describe）在体系结构和详细设计两阶段反复出现——详细设计就是把架构决策递归精化到可直接编码的粒度。

### 详细设计的输入

| 来自需求工程（RE） | 来自体系结构 |
|------|------|
| 用例 → 控制器/操作 | 模块的规格 |
| 领域模型/概念类图 → 实体类 | 导入/导出接口 |
| 系统顺序图 → 类之间协作 | 风格约束（分层/MVC/微服务）|
| 状态图 → 状态机实现 | 构件间接口 |
| 非功能需求 → 算法选择、并发策略 | 集成策略 → 依赖方向 |

> 桥接类比：需求规格（"5 分钟内 100km/h 通过"）+ 体系结构（"斜拉桥风格"）→ 详细设计（"拉索根数、路面材质"）。

### 详细设计的输出

| 视角 | 产出物 | UML |
|------|------|------|
| 静态结构 | 模块内部的类、接口与关系 | 类图、包图 |
| 动态行为 | 类之间协作与消息传递 | 顺序图、通讯图 |
| 状态行为 | 复杂对象生命周期 | 状态图 |
| 实现注解 | 构造方法、枚举、常量、静态方法 | 文字+代码 |
| 质量论证 | 为何这样设计（耦合/内聚/扩展性）| 设计理由 Rationale |

> 详细设计输出 ≠ "画几张图"，而是为后继开发者提供足够、清晰、可论证的实现蓝图。

## PART 2：面向对象详细设计——职责与协作

> 两大核心概念：**职责（Responsibility）**——类要做什么、维护什么；**协作（Collaboration）**——类之间如何通过消息一起完成大职责。
> Rebecca Wirfs-Brock："对象必须协作，否则程序将只由一个什么都做的大对象组成。"

- **职责**：操作职责（由方法实现）+ 数据职责（由属性实现）。**职责驱动分解**可能不同于功能分解。
- **委托（Delegation）**：一个对象保留高层职责，把具体执行委托给更专门的对象 = 解耦 + 关注点分离。
- **协作**：沿对象网络关系流动的消息模式，分布在对象网络中，不存在于单一位置。

### OO 详细设计过程

```
设计模型建立
├── 通过职责建立静态模型（抽象类的职责/关系，添加辅助类）
└── 通过协作建立动态模型（顺序图，明确对象创建 GRASP-Creator，选择控制风格）
设计模型重构
├── 模块化重构（高内聚、低耦合）
├── 信息隐藏重构（隐藏职责与变更）
└── 设计模式重构（GoF 23 / DDD）
```

### 静态模型：类之间的关系

| 关系 | 短语 | 解释 | UML |
|------|------|------|------|
| 普通关联 | A has a B | A 长期持有 B 引用 | ── |
| 聚合 | A owns B | 集合所属，可共享 | ──◇ |
| 组合 | B is a part of A | 整体存在则部分必存在，生命周期相同 | ──◆ |
| 继承 | B is A | 子类继承父类接口与实现 | ──▷ |
| 实现 | B implements A | 类实现接口 | ╌╌▷ |

> 选择关系时优先考虑生命周期和共享性：**能用关联就不用聚合，能用聚合就不用组合**。

### 动态模型

- **顺序图**：表示对象之间如何通过消息传递完成较大职责（消息编号 1 → 1.1 → 1.1.1）。
- **状态图**：描述对象生命周期内的离散状态及事件驱动的转移。转移格式 `event [guard] / action`；状态三类内部行为 `entry/`（进入）`do/`（持续）`exit/`（离开）。例：自动售货机（空闲→接收货款→出货中→找零）。

## PART 3：GRASP 模式

> GRASP = General Responsibility Assignment Software Patterns（Craig Larman）。不是"设计模式"，而是对象设计的**基本原则**，聚焦：把职责分配给类。

| # | 模式 | 解决的问题 |
|--|------|------|
| 1 | Low Coupling | 降低模块之间的耦合 |
| 2 | High Cohesion | 保持模块职责的聚焦 |
| 3 | Information Expert | 把职责分配给"拥有信息"的类 |
| 4 | Creator | 由谁负责创建对象 |
| 5 | Controller | 由谁负责处理系统事件 |

> **经验法则**：不存在"唯一正确"的设计，只有候选方案之间的权衡。用**耦合、内聚、稳定性**三大标尺评价，考虑未来演化压力。

- **Low Coupling & High Cohesion**：是两个元原则。若 B 已耦合于 A，则把新职责也给 A（不引入额外依赖）；一个类只承担紧密相关的职责。其余模式的取舍最终都以"是否提升内聚/降低耦合"为评判标准。
- **Information Expert（信息专家）**：把职责分配给拥有完成它所需信息的类。例：`Sales.total()` 委托给 `SalesLineItem.subtotal()`，后者调 `Commodity.getPrice()`——"谁有数据，谁负责"，自然形成职责链。
- **Creator（创建者）**：由 B 创建 A，当 B 聚合/包含/记录/紧密使用 A，或持有 A 的初始化数据（**优先聚合或包含**）。例：Sale 包含 SalesLineItem（组合）→ Sale 负责创建。
- **Controller（控制器）**：处理系统事件的职责。4 种：façade controller（Store/POST 整体系统）、role controller（Cashier 领域角色）、use case controller（BuyItemsHandler 人工类）。
> Controller 反例：表现层直接调领域对象（业务逻辑泄漏、不可复用/测试）；正例：表现层只发系统事件消息，由 Controller 协调领域对象。

## PART 4：控制风格

> 控制风格 = 系统行为如何在对象网络中分布。

| 风格 | 特征 | 评价 |
|------|------|------|
| 集中式 Centralized | 少数控制器记录所有系统行为逻辑 | 易找/易改决策；但控制器易臃肿、增加耦合、破坏信息隐藏 |
| 委托式 Delegated | 少数控制器做主要决策，其余委托专门对象 | 平衡清晰性与灵活性，**大多数实际系统采用** |
| 分散式 Dispersed | 系统行为广泛分散在对象网络 | 控制流难追踪、耦合高、内聚差，**几乎都应避免** |

| 场景 | 推荐风格 |
|------|------|
| 用例步骤清晰、状态机驱动 | Centralized |
| 业务复杂、多子领域协作 | Delegated |
| 简单 CRUD、消息频繁 | Delegated |
| 完全 P2P 对象交互 | Dispersed（慎用） |

## PART 5：模块化与信息隐藏

模块化思想发展：Wirth/Parnas（逐步求精、信息隐藏）→ Stevens（耦合/内聚定义）→ OO 模块化 → 工程实践度量。

- **Wirth 1971 逐步求精**：尽可能分解决策、推迟表示细节决策；模块化程度决定程序适应变更的难易。
- **Parnas 1972 信息隐藏**（软件工程史上最有影响的论文之一）：
  > 模块是**职责分配**而非子程序。分解准则 = **信息隐藏**：每个模块隐藏一个对其他模块保密的设计决策。
  > 模块化目标：① 模块可在对其他模块知之甚少时编写；② 模块可重组/替换而无需重组整个系统。
  > 模块化的本质：让每个模块封装一个"会改变的设计决策"——决策改变时只需改一个模块。
- **Stevens 1974 结构化设计**：耦合（模块间关联强度）、内聚（模块内元素相关程度）。内聚 6 等级（弱→强）：偶然→逻辑→时间→通信→顺序→**功能（追求最强的功能内聚）**。

### 信息隐藏基础原则 P1–P4

| 编号 | 原则 | 含义 |
|------|------|------|
| P1 | 全局变量有害 | 全局变量穿透所有模块，是隐藏的反义词 |
| P2 | 优先显示表达 | 用清晰接口/命名/文档显式表达约束，不靠隐式约定 |
| P3 | 不要重复（DRY） | 同一决策不在多处编码 |
| P4 | 面向接口编程 | 客户端依赖接口（抽象），而非具体实现 |

> 这 4 条全部源自同一目标：让"会变的部分"集中在一个模块里，对外只暴露稳定的接口。

## PART 6：为类间协作开发集成测试

> 针对复杂协作做集成测试（自顶向下或自底向上）。**Mock Object** 比 Stub 更轻量——只返回固定值用于测试，让我们只关心被测协作的逻辑，把外部依赖替换为可控的"假对象"。

```java
public class MockCommodity extends Commodity {
    double price;
    public MockCommodity(double p) { price = p; }
    public double getPrice() { return price; }
}
// 测试 sale.total() 是否 = 50*2 + 40*3 = 220
assertEquals(220, sale.total());
```

## PART 7：AI 增强——LLM 辅助详细设计

> **能力规律**：LLM 在**局部、有约束的生成任务**（单类图 88% 正确率、标准设计模式实现）表现良好；**跨类语义推理、多重关系建模、架构级权衡**是核心弱点。

| 能力维度 | LLM 表现 |
|------|------|
| 单类 UML 类图 | 88% 元素正确率 |
| 多类关系类图 | 语义错误显著增多 |
| SOLID 检测（小方法） | OCP 71.2%、DIP 64.8% |
| SOLID 检测（复杂代码） | 每 +5 LOC，F1 下降约 15% |
| 修复简单代码异味 | ~62% |
| 修复架构级异味 | 47.7%，63.1% 误报 |
| 库级系统设计 | 完成率仅 3%（函数级 82%）|

> 结论：LLM 是高效的"设计初稿生成器"，但涉及语义正确性、模块化约束、架构级权衡时，人类设计师不可缺。

**推荐工作流**：① 人类主导（职责边界识别 GRASP）→ ② LLM 生成（类图骨架/顺序图初稿）→ ③ 人类验证（语义/多类关系/耦合）→ ④ LLM 辅助（简单异味修复/Javadoc/测试）→ ⑤ 人类决策（架构级重构/信息隐藏/设计理由）。

> **关键 Prompt 技巧：给 LLM 约束，而非只给需求**（如"每个类只有一个变化原因 SRP""类之间只通过接口依赖 DIP""不暴露集合类型字段"）。加入设计约束可将代码异味减少最高 89%。

## 详细设计文档与评审

文档结构：引言（目的/词汇表/参考）→ 中层设计（静态结构+动态行为、接口规范、行为图、实现注解、设计原理）→ 低层设计（关键数据结构与算法）。

**16 项评审清单**三类：基本（一致性/详细程度/多视角）、设计考量（标准技术/简洁性/可维护/可重用/低复杂度）、过程考量（覆盖需求/遵循架构决策/详细程度足够）。

**验证**：评审（走查/检查清单/形式化 Inspection/度量）+ 测试（协作测试/Mock-Driven/变化测试 Mutation）。度量：耦合度 Ca/Ce、不稳定性 I、抽象度 A、主序列距离 D=|A+I−1|。

## 课堂练习

- **Monopoly**：掷骰子→Die（信息专家）；按 name 找 Square→Board（聚合 Square）；回合调度→MonopolyGame（Controller）；持久化→新建 IO 类（Pure Fabrication）。
- **switch-case 反模式**：`Square.landOn()` 用 switch 判断 Type，新增类型要改它，违反 OCP。正解：建行为继承树，用**多态代替 switch-case**。

## 本讲总结

> 详细设计 = 中层 + 低层；**职责 + 协作**是 OO 详设两大支柱；GRASP（Expert/Creator/Controller/Low Coupling/High Cohesion）指导职责分配；控制风格分集中/委托/分散；Parnas 信息隐藏——每个模块封装一个"会变的设计决策"；集成测试用 Mock 验证协作。
