# 04-02 面向对象的模块化与信息隐藏

> 软件工程与计算 II · 详细设计模块 · 第 2 讲
> **本讲主题**：访问耦合 · 继承耦合 · 信息隐藏 · 为变更而设计
> **原则编号体系**：P1–P4 见 04-01（信息隐藏基础），P5–P14 见本讲（面向对象设计）

## PART 1：模块化基础与耦合概念

> 模块 = 可独立设计、实现、测试的自包含软件单元。OO 中三种粒度：方法级、类级（状态+行为）、包级。
> 两个核心指标：**耦合**（模块间依赖强度，越低越好）+ **内聚**（模块内元素关联程度，越高越好）。目标：**高内聚、低耦合**。

OO 系统中耦合来自两个方向：
- **访问耦合（Component Coupling）**：A 使用 B 的服务（调用方法/访问字段/参数传递/消息链）——"黑盒复用"，只依赖接口。
- **继承耦合（Inheritance Coupling）**：A 继承 B，子类依赖父类实现细节——"白盒复用"，更深更隐蔽。

## PART 2：访问耦合

### 五个级别（从高到低）

| 级别 | 名称 | 说明 |
|------|------|------|
| 5（最高）| 隐式访问耦合 | 依赖全局变量或 Singleton |
| 4 | 实现中访问耦合 | 访问另一类私有实现细节（反射 private） |
| 3 | 成员变量访问耦合 | 直接读写另一类的 public 字段 |
| 2 | 参数变量访问耦合 | 通过方法参数传递，依赖具体类型 |
| 1（最低）| 无访问耦合 | 无直接依赖 |

> 级别 3–5 打破封装，应重构为通过方法（消息）访问；级别 1–2 可接受。

### Cascading Message（消息链/Train Wreck）

```java
// 问题：customer.getAddress().getCity().toUpperCase() —— 三层实现细节全暴露
// 方案 B（推荐）：委托，调用方只需知道 Customer
String city = customer.getCityName();
```

### P5 迪米特法则（Law of Demeter）

> **只与你的直接朋友交谈。** 一个方法只能调用：自身(this)、方法的参数、方法内新创建的对象、类的成员变量（直接持有）。是对访问耦合的直接约束——避免消息链。

### P4 面向接口编程 & P6 接口隔离原则（ISP）

```java
List<String> list = new ArrayList<>();  // 依赖接口而非具体类
```

> **P6 ISP**：客户端不应被迫依赖它们不使用的接口。肥接口（`Worker` 含 work/eat/sleep，Robot 不需要 eat/sleep）应拆分为 `Workable` + `Feedable`，调用方只依赖真正需要的接口。

## PART 3：继承耦合

### 四种继承类型

| 类型 | 说明 | 耦合 | 推荐度 |
|------|------|------|------|
| Modification（修改）| 子类覆盖父类方法并改变语义 | 最高 | ✗ 避免 |
| Refinement（精化）| 子类调 super 前后加行为，不改语义 | 中 | △ 谨慎 |
| Extension（扩展）| 子类只新增方法，不覆盖 | 低 | ✓ 安全 |
| Nil（无实现）| 父类是纯抽象接口，子类提供实现 | 最低 | ✓✓ 最佳 |

> 推荐优先使用抽象父类（Nil）和扩展继承（Extension）。

### P7 里氏替换原则（LSP, Liskov 1987）

> **子类型必须能够替换其父类型而不改变程序的正确性。**

经典反例：正方形 ≠ 长方形（Square 的 setWidth 同时改 height，破坏 Rectangle 的"宽高独立"不变式）；企鹅 ≠ 会飞的鸟（Penguin.fly() 抛异常）。修复：用接口分离（Flyable）代替强制继承。

> **LSP 的契约含义**：子类**前置条件放宽**（≤父类）、**后置条件加强**（≥父类）、**不变式维持**。记忆口诀："子类前置放宽，后置加强，不变式维持——这才叫替换。"

### P8 组合优于继承（GoF 1994）

| 继承的问题 | 组合的优势 |
|------|------|
| 白盒复用，依赖父类实现 | 黑盒复用，只依赖接口 |
| 父类修改可能破坏所有子类 | 运行时可动态切换组合对象 |
| 多维度变化时层次难扩展 | 职责分离更清晰 |
| 强迫 IS-A 语义 | 无需 IS-A 约束 |

> 例：`Stack extends Vector`（历史错误）→ `Stack` 内部持有 `Vector`（组合）。

## PART 4：面向对象的信息隐藏

> **每个模块应对其他模块隐藏一个重要的设计决策。**（Parnas 1972）

**两类秘密**：主要秘密（职责变更的可能性——这个类应该做什么会变吗？）+ 次要秘密（实现变更的可能性——这个类如何做会变吗？）。
> 变化是软件的常态。**信息隐藏 = 隔离变化 = 降低耦合。**

**类的职责 = 状态（State，字段）+ 行为能力（Behavioral Capability，方法）**。类的接口 = 对外承诺的行为；类的实现 = 内部状态与算法细节。

### 封装的五种类型（A–E）

| 类型 | 隐藏内容 | 核心手段 |
|------|------|------|
| A — 数据类型（ADT）| 内部数据表示 | 访问器/修改器替代 public 字段 |
| B — 内部结构 | 数据的组织结构 | 不暴露内部容器/数组 |
| C — 其他对象引用 | 内部持有哪些对象 | 不直接返回内部对象引用 |
| D — 类型信息（LSP）| 对象的具体运行时类型 | 面向接口，多态调度 |
| E — 潜在变更 | 可能变化的实现策略 | 策略模式、依赖注入 |

> A–C 处理当前实现细节的封装；D–E 处理未来变化的封装。

- **类型 A（ADT）**：`Point` 私有 x/y，对外提供直角坐标接口（getX/getY）和极坐标接口（getR/getTheta）——同一 ADT 改变内部实现不影响调用方。用**访问器/修改器**替代 public 字段（可加验证、懒加载、计算）。
- **类型 B/C**：用 **Iterator 模式** 或 `Collections.unmodifiableList()` 隐藏内部集合，受控的 addStop() 可加业务验证。调用方只知可遍历，不知内部是 List/Set/数组。
- **类型 D**：用**多态调度**代替 `instanceof` 判断（每增一种 Shape 不用改 process 代码）。
- **类型 E**：用**策略模式**封装变更（Sorter 委托给 SortStrategy），与**开闭原则（OCP）**直接对应。

## PART 5：为变更而设计（P9–P14）

### P9 为变更而封装

> 找到系统中变化的部分，将其封装起来，使变化不影响不变的部分。

识别变化点三问：哪些需求最可能改变？哪些部分跨边界共享？哪些已变过多次（历史是最好的指示）？
封装手段：算法/策略→策略模式；对象创建→工厂方法；对象结构→组合模式；数据格式→Adapter。

### P10 权限最小化（Effective Java, Bloch）

> 最小化类和成员的可访问性。Java 四级别：private → package-private → protected → public。
> **从最严格的 private 开始，只在确实需要时才放宽。** 每增加一个 public 成员就是一个难以撤回的对外承诺。好处：接口更小→耦合更低、内部可自由修改、降低维护成本、提高安全性。

### P11 开闭原则（OCP, Meyer）

> 软件实体应**对扩展开放，对修改关闭**。把 `if/else` 类型判断重构为 `DiscountPolicy` 接口 + 多个实现类，新增类型只需新增实现类。

### P12 依赖倒置原则（DIP）

> 高层模块不应依赖低层模块，两者都应依赖抽象；抽象不应依赖细节，细节应依赖抽象。`BusinessLogic` 依赖 `Database` 接口（构造器注入），而非 `MySQLDatabase` 具体类。

### P13 单一职责原则（SRP, Martin）

> 一个类应**只有一个变化原因**。把混了"薪资计算+持久化+报表"三个变化原因的 `Employee` 拆为 PayCalculator/EmployeeRepository/EmployeeReport。
> SRP 与信息隐藏直接关联：每个类只隐藏一个变化原因（= Parnas 的"设计决策"）。违反症状是 **God Class**。

### P14 SOFA（方法级设计原则）

> **S**hort（方法≤20–30 行）· **O**ne thing（一个短句能描述，需"and"则拆分）· **F**ew arguments（>3 个参数考虑参数对象）· **A**bstraction level consistency（不要高层业务逻辑混入低层 db.execute）。与 SOLID 的类粒度原则互补。

### 原则汇总记忆结构

> **P1–P4 防止"隐式依赖"；P5–P8 控制"耦合来源"；P9–P13 管理"变化封装"；P14 约束"方法粒度"。**

## PART 6：LLM 与面向对象设计

> "当前 LLM 是有效的辅助设计工具，但不是自主设计主体。"

| 维度 | LLM 表现 | 原因 |
|------|------|------|
| 内聚分析 | 较稳定 | 局部信息足够 |
| 耦合推理 | 噪声下崩溃（F1 降幅 >50%）| 需跨类全局推理 |

- **量质反比定律**：代码体积几乎是结构退化的完美预测因子。LLM 生成代码比人类基线多 42%–85% 代码异味。模型能力越强→代码越长→结构越臃肿。
- **模块化悖论**：即使明确给模块化指导，LLM 仍倾向把新功能加入已有类而非新建职责单一的类 → 默认违反 SRP。
> 根本原因：LLM 训练目标是预测下一个 token，感知不到跨文件的全局模块化约束。

**代码异味修复能力**：简单异味（Magic Number、Empty Catch）~62% 可修复，适合 CI 自动修复；架构级异味（Broken Modularization、Deficient Encapsulation）显著劣于人类，SmellBench 2026 最佳解决率仅 47.7% 且 63.1% 误报。

**SOLID 检测实证（"Are We SOLID Yet?" 2025）**：OCP 71.2%、DIP 64.8% 最适合 LLM（违反模式局部、规律性强）；SRP/LSP 需跨上下文语义理解，复杂类体系准确率接近随机。**每 +5 LOC，F1 下降约 15%**。结构化推理步骤比参数规模更关键（小模型+好 Prompt > 大模型+零样本）。

工具推荐：**iSMELL**（LLM + 7 专家静态分析工具，God Class/Feature Envy 检测 F1 比纯 LLM +35%）。

### 如何正确使用 LLM 辅助

**有效用法**：生成单类骨架（带信息隐藏约束）、生成 PlantUML 类图初稿、识别简单代码异味、封装五类型快速诊断、检测小方法（≤30 行）的 OCP/DIP 违反。

**危险/不可依赖**：多类关系设计（多重度错误、关联方向反转需逐条审查）、跨类耦合评估、职责边界划分（LLM 不知道"什么会变"）、库级全系统模块化设计（全任务完成率仅 3%）。

> **最佳实践**：把 LLM 当"知识渊博但不了解你业务的初级设计师"——它熟悉 SOLID/GoF/封装，但需要你提供领域约束和变化点，才能生成有价值的初稿。**人类保留职责边界划分、架构级重构、设计理由的最终决策权。**

## 小结

| 模块化 | 信息隐藏 |
|------|------|
| 耦合 = 模块间依赖强度（越低越好） | 每个模块隐藏一个设计决策（P13 SRP）|
| 访问耦合 5 级别：隐式>实现>成员>参数>无 | 类职责 = 状态 + 行为能力 |
| 消息链破坏 P5 迪米特法则 → 委托 | 封装 A(ADT)/B(结构)/C(对象)/D(类型)/E(变更) |
| P6 ISP：接口不强迫客户端依赖不需要的方法 | Iterator 隐藏集合结构 |
| 继承耦合：Modification>Refinement>Extension>Nil | P10 权限最小化：从 private 开始 |
| P7 LSP：子类前置放宽、后置加强、不变式维持 | P11 OCP / P12 DIP 是封装变更的架构级表达 |
| P8 组合 = 黑盒复用，更灵活 | — |

> **高内聚、低耦合——软件模块化设计的永恒目标。**
