## PART 1：从一次部署事故开始

### Knight Capital · 45 分钟 · 4.4 亿美元蒸发

| 项 | 内容 |
|----|------|
| 时间 | 2012-08-01 09:30，新交易系统上线 |
| 经过 | 45 分钟内以错误价格持续买卖股票，亏损 4.4 亿美元 |
| 调查结论 | 8 台交易服务器中 1 台未更新，仍运行含旧 "Power Peg" 代码的旧版本 |
| 根本原因 | 部署流程无检查清单，配置状态未验证，旧代码未彻底禁用 |
| 结局 | 公司被迫出售，实际消失 |

> 不是算法错误，不是网络故障——是一次**配置管理的彻底失效**。系统无法自动发现 7 台新、1 台旧的不一致。

## PART 2：项目管理基础

### 什么是"项目"？

> 项目是为创建唯一产品、服务或成果而进行的**临时性**工作。——PMBOK

| 要素 | 说明 | 与"日常运营"的区别 |
|------|------|------|
| 临时性 | 有明确起止时间 | 运营是持续、无终止的 |
| 唯一性 | 产出物独特、不完全重复 | 运营是标准化重复流程 |
| 渐进明细 | 随进展逐步明确 | 运营流程预先固化 |

```
        范围 Scope
         / \
        /   \
       / 质量 \
      /________\
   时间 Time   成本 Cost
```

> 项目管理目标：在**铁三角**约束下交付质量。五大过程组：启动 · 计划 · 执行 · 跟踪控制 · 收尾。

### 三种团队结构

团队是为了一致的目的、绩效标准、方法而共担责任并且技能互补的少数人。
| 结构 | 决策方式 | 优点 | 缺点 | 适用 |
|------|------|------|------|------|
| 主程序员团队 | 集中，主程序员定一切 | 效率高·协调少 | 主程序员瓶颈·积极性低 | 小型精英团队 |
| 民主团队 | 分散，共同决策 | 士气高·创意多 | 决策慢·职责模糊 | 研究探索性项目 |
| 开放团队 | 混合，正式+非正式 | 灵活·平衡 | 管理复杂·需强沟通 | 中大型工程项目 |

> 现代普遍选择**开放团队**。典型实例 Scrum：Product Owner（需求优先级）+ Scrum Master（保障流程）+ Dev Team（自组织开发）。

### 风险管理：项目管理的第四维度

> 风险 = 不确定事件 × 影响程度。流程：识别 → 分析 → 优先级排序 → 应对 → 监控（持续循环）。

| 风险类型 | 示例 | 应对策略 |
|------|------|------|
| 技术风险 | 新框架不成熟 | 原型验证(Spike)、技术预研 |
| 人员风险 | 核心成员离职 | 知识共享、交叉培训(Bus Factor ≥ 2) |
| 需求风险 | 范围蔓延 | 迭代交付、变更控制 |
| 进度风险 | 低估工作量 | 缓冲时间、关键路径分析 |
| 外部风险 | 第三方 API 变更 | 抽象层隔离、合规审查 |

> Knight Capital 中技术风险（旧代码未清理）和进度风险（赶在开盘前部署）同时爆发。

### AI 对团队的深层影响

- **10× 个体生产力**：AI Coding Agent 使单人完成过去小组的代码量。
- **Bus Factor 悖论**：降低对个人知识的依赖，但增加对 AI 工具链的依赖。
- **新技能需求**：Prompt Engineering、AI Output Review、系统设计能力比编码更重要。
- **管理者新角色**：从"分配任务"转向"设计人机协作流程"。

> 当 AI 可以写代码时，工程师的核心价值是**问题定义、架构决策和质量判断**。

## PART 3：软件质量保障（SQA）

### 质量的两个维度

| 维度 | 定义 | Knight Capital 体现 |
|------|------|------|
| 符合规格 | 满足需求规格说明书 | 新系统"符合规格"——但部署不符合规格 |
| 用户满意 | 使用中感受到价值 | 交易对手 45 分钟内感受到"不满意" |

> 质量不只是代码质量——**部署过程本身也是质量的一部分**。

### 软件质量保障 SQA 主要活动

```
代码评审（人工审查：逻辑·安全·规范）
    ▼
静态分析（工具自动检测：缺陷·规范·复杂度）
    ▼
动态测试（运行验证：单元·集成·系统测试）
    ▼
过程审计（检查开发与发布过程是否合规）← Knight Capital 最缺这步
```

> **评审 vs 测试**：评审发现"代码写了什么"，测试发现"代码做了什么"。二者各有盲区，缺一不可。

### 质量模型（ISO/IEC 9126）

从众多质量属性定义中选择相互配合、相互联系的**特征集**，称为**质量模型**

六大质量因素：功能性、可靠性、易用性、效率、可维护性、可移植性。

> 质量度量（DeMarco 1998）："你不能控制自己无法度量的东西。" 测度(Measure)→测量(Measurement)→度量(Metric)。

**评审过程六阶段**：规划 → 总体部署 → 准备 → 审查会议 → 返工 → 跟踪。

### AI 时代质量新挑战

| 传统 | AI 时代 |
|------|------|
| 人写代码 → 人审查 | AI 生成代码 → 人审查 AI 输出 |
| 缺陷来源：人的疏忽 | 缺陷来源：幻觉·过时知识·上下文丢失 |
| 测试覆盖率是金标准 | 覆盖率 + 语义正确性验证 |

**AI 生成代码的质量风险**：幻觉代码（调用不存在的 API）、安全漏洞、版权风险、技术债务、过度自信。

> 核心原则：AI 是质量保障的"加速器"而非"替代者"——**最终质量判断权必须在人手中**。

## PART 4：软件配置管理（SCM）

### 四大活动

| 活动 | 核心问题 | Knight Capital 缺失了什么 |
|------|------|------|
| 配置项确定 | 哪些制品需版本控制？ | 服务器配置状态未纳入管控 |
| 配置控制 | 变更如何审批执行？ | 部署无标准流程，无逐台确认 |
| 状态报告 | 各节点版本什么状态？ | 第 8 台服务器状态不可见 |
| 配置审计 | 实际配置是否符合规定？ | 上线前未做一致性审计 |

软件开发中产生大量**中间制品**（需求规格、设计模型、源码等），是不同阶段协同的基础
制品变更的最大挑战：确保使用者获得最新的制品，避免**协同问题**

> **配置项(CI)**：纳入配置管理的各种项目。**基线(Baseline)**：已正式评审的规格或产品，只有通过正式变更控制才能变更（需求基线/体系结构基线/详细设计基线）。

### 版本控制两种策略

- **lock-modify-unlock**（锁定-修改-解锁）：简单安全，但串行低效。
- **copy-modify-merge**（复制-修改-合并）：并行效率高，需解决冲突。**现代 Git 采用此策略**。

### 变更控制：Knight Capital 缺失的环节

```
变更请求(CR) → 影响分析 → 风险评估 ← Knight Capital 跳过了这一步
    → CCB 评审(批准/拒绝/回滚预案) → 执行变更(逐节点验证) → 上线验证(一致性检查)
```

> 软件部署是一种配置变更，必须遵守变更控制流程。"赶进度跳过流程"的代价可能是公司本身。

### Git 三大分区与分支策略

```
工作区 ──git add──→ 暂存区(Index) ──git commit──→ 版本库(Repo) ──git push──→ 远程仓库
```

| 分支 | 用途 | 保护规则 |
|------|------|------|
| main | 生产稳定版 | 禁止直接 push，只接受合并请求 |
| develop | 持续集成 | 合并前必须通过 CI |
| feature/xxx | 功能开发隔离 | 完成后合并到 develop |
| hotfix/xxx | 生产紧急修复 | 合并到 main 后同步到 develop |

> 仅有 main、所有人直接提交 → 版本混乱、回滚困难。**分支策略是团队协作的工程纪律**。

### 从 SCM 到现代演进

```
传统SCM(2000s) → DevOps(2010s) → Platform Engineering(2020s)
版本控制         CI/CD自动化管线   内部开发者平台(IDP)
手动构建发布      基础设施即代码IaC  Golden Path 标准化
```

- **CI/CD**：每次部署都自动化、可重复、可验证——不再依赖人工逐台检查。
- **部署策略**：蓝绿部署（秒级回滚）、金丝雀发布（控制爆炸半径）、滚动更新（零停机）、特性开关（解耦部署与发布）。
- **IaC**：用代码描述基础设施，"期望状态"只定义一次，系统自动保证一致——若 Knight Capital 用 IaC，8 台配置不可能不一致。
- **GitOps**：Git 是唯一事实源、声明式而非命令式、自动同步、可审计。
- **LLMOps = DevOps + ML 模型管理 + Prompt 工程**：模型版本、Prompt 版本、AI Pipeline 配置成为新配置项。

## PART 5：今天最重要的四个结论

1. **团队结构决定协作模式，没有万能答案**——Scrum 开放团队适合大多数现代项目。
2. **质量不只是"代码没 Bug"**——SQA 覆盖整个开发与发布过程，发布过程本身也是质量对象。
3. **配置管理是团队协作的工程纪律底线**——1 台服务器配置不一致 = 4.4 亿美元 + 一家公司。
4. **AI 改变了效率，但不改变工程纪律的本质**——变更速度更快、影响面更大，纪律更重要。

> "配置管理是工程纪律，不是可选项——Knight Capital 用 4.4 亿美元证明了这一点。"
