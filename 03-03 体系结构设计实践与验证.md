# 03-03 体系结构设计实践与验证

> 软件工程与计算 II · 体系结构模块 · 第 3 讲
> **核心三问**：① 如何将用例分析结果系统映射为包结构？ ② 如何通过接口设计与依赖倒置实现层间解耦？ ③ 如何选择集成策略并验证体系结构的正确性？

## PART 1：逻辑视角设计

> **包（Package）**是体系结构逻辑视角的基本构造单元，好的包划分直接决定系统的可维护性。目标：高内聚、低耦合。

### 从用例到包的 4 步映射

| 步骤 | 活动 | 产出 |
|------|------|------|
| 1 | 确定顶层包结构 | controller / service / dao / entity |
| 2 | 分析用例，确定业务模块 | book / cart / order / customer / payment ... |
| 3 | 将概念类分配到 entity 包 | Book, Order, Customer, CartItem |
| 4 | 为每层建立与业务模块对应的类 | BookController, BookService, BookDao |

> 原则：**先纵向分层，再横向分模块**，确保每个包职责单一。依赖方向**单向向下**：controller → service → dao → entity，禁止反向或跨层。

### 包设计六原则（Robert C. Martin）

**内聚性三原则**（哪些类放一起）：

| 原则 | 全称 | 含义 |
|------|------|------|
| REP | 重用-发布等价 | 重用的粒度 = 发布的粒度 |
| CRP | 共同重用 | 一起被重用的类放同一包 |
| CCP | 共同封闭 | 因同一原因变更的类放同一包（包级 SRP） |

> **CCP 与 CRP 互斥**：CCP 服务维护者（倾向大包），CRP 服务重用者（倾向小包）。早期以 CCP 为主（大包便于开发），后期架构稳定后重构为 CRP 主导（便于复用）。没有正确答案，只有当前阶段的权衡。

**耦合性三原则**（包之间如何依赖）：

| 原则 | 全称 | 含义 |
|------|------|------|
| ADP | 无环依赖 | 包之间的依赖不能成环 |
| SDP | 稳定依赖 | 依赖方向朝更稳定的包 |
| SAP | 稳定抽象 | 越稳定的包应越抽象 |

> **不稳定度** `I = Ce/(Ca+Ce)`（Ca 传入依赖、Ce 传出依赖）。entity（I=0 最稳定）← dao ← service ← controller（I=1 最不稳定）。**SDP+SAP = 包级别的依赖倒置（DIP）**。痛苦区=稳定但具体（要重构为接口）；无用区=不稳定但抽象（可删）。

### ADP 违规修复（循环依赖）

OrderService ↔ InventoryService 互调 → 违反 ADP。
- **方案 A**：提取共同依赖包（新建 stock-api 含 StockQuery 接口，两者都依赖它，A→C←B）。
- **方案 B**：接口定义在调用方包内（反转依赖方向）。
> 核心都是**面向接口编程**。

## PART 2：4+1 视图

> **Viewpoint（视点）**= 创建某类视图的约定规范（如建筑制图标准）；**View（视图）**= 从某视点出发的一次具体表达（一张具体平面图）。

| 视图 | 关注点 | UML 图 | 面向角色 |
|------|------|------|------|
| 逻辑视图 | 功能需求、类与包组织 | 包图、类图 | 架构师、开发者 |
| 开发视图 | 代码组织、模块划分 | 组件图 | 开发者 |
| 进程视图 | 并发、同步、性能 | 活动图、序列图 | 系统集成者 |
| 物理视图 | 部署拓扑、硬件映射 | 部署图 | 运维 |
| 场景视图 | 用例驱动、验证其他视图 | 用例图 | 所有人 |

在线书店案例：
- **逻辑视图**：顶层包内按业务模块再划分。
- **物理视图**：四层代码全部部署在同一 Tomcat 进程（单体）。
- **开发视图**：4 个 Maven 模块（web/service/dao/entity），entity 是公共依赖。
- **进程视图**：200 Tomcat worker + 20 DB 连接 + 定时器 + 异步池；库存乐观锁、订单超时扫描、支付幂等、缓存最终一致。
- **场景视图（粘合剂/验证视图）**：通过走查关键用例（如"购买图书"穿透 4 层 + 库存锁 + 支付幂等），检验四个视图是否协调一致。

> **逻辑分层 ≠ 物理分离**：单体架构中逻辑层共享同一进程。

## PART 3：接口设计与依赖倒置

> **接口 = 层与层之间的"楼板"/契约**：解耦（上层不依赖下层实现）、可替换、可测试（Mock）。先定义接口，再写实现——面向接口编程。

**PO vs VO**：

| 概念 | 全称 | 用途 | 所在层 |
|------|------|------|------|
| PO | Persistent Object | 与数据库表一一对应 | entity/dao |
| VO | Value Object | 面向展示层的数据传输 | controller/service |

> 区分原因：PO 含数据库细节（主键、外键、时间戳），VO 只含前端需要的字段，**隔离变化**（数据库字段变更不影响前端接口）。

### 依赖倒置原则（DIP）

> ① 高层模块不应依赖低层模块，两者都应依赖抽象；② 抽象不应依赖细节，细节应依赖抽象。

```java
@Service
public class BookServiceImpl implements BookService {
    private final BookDao bookDao;   // 依赖接口，不依赖实现类
    @Autowired
    public BookServiceImpl(BookDao bookDao) { this.bookDao = bookDao; } // 构造器注入
    public BookVO findById(Long id) { return convertToVO(bookDao.findById(id)); }
}
```

> **无状态 Service 设计**：Spring 默认单例，多个请求共享同一实例。Service 持有请求级状态（如 `currentBook` 字段）会在多线程下数据混乱。要点：接口=契约、PO/VO 分离、DIP、构造器注入、无状态。

## PART 4：体系结构构建（落地）

> 落地原则：① **目录结构 = 包结构** ② 依赖检查（import 只能指向允许的层）③ 接口先行（先建接口文件，再写实现）。

Spring Boot 项目结构：`controller/` `service/`（接口）+ `service/impl/` `dao/`（接口）`entity/`。配置文件（application.properties）体现物理视图：端口、数据库地址、技术选型。

## PART 5：集成策略与验证

| 策略 | 方式 | 优点 | 缺点 |
|------|------|------|------|
| 大爆炸 | 所有模块一次集成 | 简单直接 | 难定位缺陷 |
| 自顶向下 | 从 controller 开始 | 早期验证主流程 | 需大量 Stub |
| 自底向上 | 从 dao 开始 | 底层测试充分 | 需大量 Driver |
| 三明治 | 顶+底同时，中间最后 | 平衡两者 | 中间层测试较晚 |
| 持续集成 | 每次提交触发 | 快速反馈 | 需自动化基础设施 |

> **Stub（桩）**替代被调用的下层模块（用于自顶向下）；**Driver（驱动）**替代调用者的上层模块（用于自底向上）。

**CI 流程**：编译构建 → 单元测试 → 集成测试 → 代码质量检查（SonarQube）→ 部署测试环境。价值：快速反馈、降低风险（频繁小集成 vs 一次大集成）。

**评审方法**：ATAM（分析质量属性权衡，适合大型系统早期）、SAAM（基于场景，适合中小型）。二者都强调：在编码前发现架构问题，修复成本远低于编码后。

**集成测试**（`@SpringBootTest` + MockMvc）验证：层间连通性、依赖注入正确装配、事务正确传播。

### "好架构"的判断标准

> 古罗马 Vitruvius：好建筑应 firmitas（坚固）、utilitas（实用）、venustas（美观）。
> 四个评估问题：① 能否应对演化？② 是否适用于预期用途？③ 是否经济可行？④ 对维护者是否清晰可理解？

### 架构度量

| 指标 | 公式 | 含义 |
|------|------|------|
| Ca 传入依赖 | 依赖本包的外部包数 | 越高越稳定 |
| Ce 传出依赖 | 本包依赖的外部包数 | 越高越不稳定 |
| I 不稳定度 | Ce/(Ca+Ce) | 0=完全稳定, 1=完全不稳定 |
| A 抽象度 | 抽象类数/总类数 | 接口包高，实现包低 |

> DevOps 度量（变更前置时间、部署频率、MTTR、变更失败率）也是架构健康度指标。
> **ADL（架构描述语言）**：起源 MIL（C2/Rapide/ACME/Wright）；工业界 **UML 已成事实标准 ADL**。本课用 PlantUML/Mermaid/Spring Boot 目录结构。

## PART 6：AI 辅助体系结构

> **LLM 集成系统参考架构**：Interaction Layer（Prompt 管理）→ Orchestration Layer（任务编排，LangChain/CrewAI）→ Model Layer（推理）→ Data Layer（向量存储/RAG）→ Infrastructure Layer（部署/监控/Guardrails）。

**AI 能否取代架构师？** AI 能秒级生成骨架、一致性高、分析大规模依赖图；但缺乏业务上下文、无法权衡组织/政治/成本、不理解利益相关人真实关注点。
> 结论：**AI 是架构师的放大器，不是替代者。"Humans own architecture, AI validates it."**

AI 辅助包依赖分析示例：检测「dao.order → service.inventory」违反分层约束（dao 层不应依赖 service 层），应将库存查询逻辑提升到 service 层。

**审查 Checklist**：依赖方向是否单向向下？是否存在循环依赖？接口和实现是否分离？entity 包是否只含数据实体？是否遵循 CCP/CRP？
