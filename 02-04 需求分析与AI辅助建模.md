# 02-04 需求分析方法与 AI 辅助建模

> 软件工程与计算 II · 模块二 · 单元 4
> **核心三问**：① 需求分析的目的是什么，有哪些常用模型？ ② 概念类图、顺序图、状态图分别描述什么视角？ ③ LLM 能自动生成 UML 图吗，质量如何评估？

> 从用户的声音到精确的模型——需求分析让沟通变为共识。

## PART 1：需求分析基础

### 为什么要需求分析

```
需求获取结果          需求分析            需求开发目标
· 用户的理解   ────  · 建立分析模型  ────  · 共同的理解
· 问题的描述         · 创建解决方案        · 解决方案的描述
```

> 需求分析是从**获取结果到共同理解的桥梁**。核心目标：消除歧义、建立模型、为设计奠基。

**两个任务**：① 建立分析模型（整理提炼获取信息，达成开发者与用户的共同理解）；② 创建解决方案（依据共同理解发挥创造性）。

> 分析模型是开发者与用户之间的"共同语言"。

### 建模手段

> "模型是对事物的抽象，帮助人们在创建一个事物之前可以有更好的理解" —— Blaha 2005

- **抽象（Abstraction）**：忽略细节，聚焦本质特征和关键关系。
- **分解（Decomposition）**：将复杂系统拆分为可理解的子部分。

> 好的模型 = 合理的抽象层次 + 恰当的分解粒度。**建模不是目的，沟通和理解才是目的。**

### 常见分析模型对照

| 方法 | 模型 | 描述 |
|------|------|------|
| 结构化 | 数据流图 DFD | 从数据传递加工角度描述功能处理 |
| 结构化 | 实体关系图 ERD | 描述数据对象及其关系 |
| 面向对象 | 用例图 | 描述用户与系统交互、系统边界 |
| 面向对象 | 类图 | 描述领域概念及关系，捕获静态结构 |
| 面向对象 | 顺序图 | 描述一次交互的行为过程、对象协作 |
| 面向对象 | 状态图 | 描述对象生命期内的状态变化 |

> 本课程重点关注**面向对象方法**的分析模型。

**OO 分析三个关键步骤**：① 细化系统对外交互（用例图→用例描述）→ ② 明确协作对象（概念类图）→ ③ 明确协作行为（顺序图/状态图）。

## PART 2：概念类图

### 定义

描述应用领域中重要的概念及其关系，捕获系统**静态结构**，也称"领域模型"。

| 对比 | 概念类图（分析阶段） | 设计类图（设计阶段） |
|------|------|------|
| 关注点 | 问题域中的概念和关系 | 软件内部的类和接口 |
| 方法 | 通常不含方法实现细节 | 含方法签名和实现逻辑 |
| 来源 | 从用例描述和领域知识提取 | 从概念类图映射而来 |

### 建立过程（四步）

① 识别概念类（从用例描述提取**名词**，筛选有意义的概念）→ ② 确定关联关系（识别概念间的"有关系"）→ ③ 添加属性 → ④ 确定多重性（标注 `1..*`、`0..1` 等）。

> 名词提取法是最常用的方法：名词→概念类，动词→关联关系，数量词→多重性（"至少一个"→`1..*`）。

### 候选类向概念类的转化

| 候选类特征 | 处理方式 |
|------|------|
| 既有状态，又有行为 | 确定为概念类 |
| 有状态，无行为 | 作为其他概念类的属性 |
| 无状态，有行为 | 先审视需求是否遗漏；确认无遗漏则剔除，行为转交有状态的类 |
| 无状态，无行为 | 完全剔除 |

> **核心判据：状态是对象存在的基础，行为需要状态的支撑。**

筛选案例（超市 POS）：会员（有状态有行为→概念类）、销售（→概念类）、顾客（无关联→剔除）、收据（无状态无行为→剔除）、商品标识（→作为商品类的属性）。

**识别关联三来源**：① 分析用例文本（发现协作）② 分析问题域关系（补充整体-部分关系）③ 精简（去除冗余/导出关联）。
**识别属性**：属性往往是类协作时必要的信息（条件、输入、结果、过程记录）。需合并所有局部概念类图才能全面体现。

## PART 3：顺序图

描述系统中**一次交互的行为过程**，展示对象间消息传递的时间顺序，从用例描述的正常流程映射而来。

**核心元素**：参与者(Actor)、生命线(Lifeline)、消息(Message)、激活框(Activation)。

### 从用例描述到顺序图（映射规则）

| 用例描述 | 顺序图对应 |
|------|------|
| 参与者的动作 | 从参与者到系统的消息 |
| 系统的响应 | 从系统到参与者的返回消息 |
| 每一步交互 | 一条消息箭头 |
| 交互的时间顺序 | 从上到下排列 |

> 正常流程先画，扩展流程用 `alt` / `opt` 片段补充。

## PART 4：状态图

描述系统、用例或对象在整个**生命期内的状态变化**，也称"状态机图"。

**基本元素**：状态(State，圆角矩形)、转换(Transition，箭头)、事件(Event，转换标签)、动作(Action，`/action`)、初始状态(实心圆)、终止状态(圆中圆)。

> 状态图关注**单个对象在不同事件驱动下的行为变化**，适合生命周期复杂的实体：
> - 订单：待支付 → 已支付 → 已发货 → 已完成
> - 设备：空闲 → 运行中 → 故障 → 维修中
> - 用户会话：未登录 → 已登录 → 超时 → 已注销

> **三大视角小结：类图看静态结构，顺序图看交互行为，状态图看生命周期。**

## PART 5：AI 辅助 UML 图生成

### LLM + UML 工具链

| 工具 | 特点 | LLM 配合方式 |
|------|------|------|
| PlantUML | 覆盖全部 UML 图类型，语法严谨 | LLM 生成代码 → 渲染引擎出图 |
| Mermaid | JS 原生，GitHub/Markdown 内嵌 | LLM 生成代码 → 浏览器直接渲染 |
| Claude Artifacts | 直接生成交互式可视化 | 对话式生成，即时预览 |

> 典型工作流：需求描述（自然语言）→ LLM 生成代码 → PlantUML/Mermaid 渲染 → 人工审查修正。

### LLM 生成各类 UML 图的质量评测

| 图类型 | LLM 表现 | 关键问题 |
|------|------|------|
| 用例图 | 较好 | include/extend 关系容易误用 |
| 类图 | 中等 | 关联关系和多重性判断不准确 |
| 顺序图 | 中等偏下 | 消息顺序、激活框处理差 |
| 状态图 | 较弱 | 状态转换条件容易遗漏 |

> 关键发现：**图越大越复杂，质量越差**——建议拆解为小的、聚焦的组件；不同 LLM 表现有差异；所有 LLM 生成的 UML 都需人工审查。

### MermaidSeqBench（IBM Research · NeurIPS 2025）

顺序图生成基准，132 个人工验证样本，**6 个评分维度**：① Syntax 语法正确性 ② Mermaid Only 语法规范 ③ Logic 逻辑正确性 ④ Completeness 完整性 ⑤ Activation Handling 激活框处理 ⑥ Error & Status Tracking 错误追踪。

> 结果：各模型在**激活框处理和错误追踪**方面存在显著能力差距。

### AI 建模五条最佳实践

① **拆解再生成**（大系统拆成小组件）② **提供明确上下文**（参与者列表、用例描述、业务规则）③ **指定输出格式**（PlantUML/Mermaid）④ **必须人工审查**（重点查关系、多重性、消息顺序、状态转换）⑤ **Few-shot 示范**（先给正确示例）。

> **AI 生成的是初稿，不是终稿。工程师的价值在于审查、修正和决策。**

## 课堂总结

| 主题 | 关键结论 |
|------|------|
| 需求分析 | 通过建模达成开发者与用户的共同理解 |
| OO 分析核心产出 | 用例图 → 概念类图 → 顺序图 → 状态图 |
| AI 辅助建模 | LLM + PlantUML/Mermaid 可快速生成 UML 初稿 |
| AI 的弱点 | 关系判断、激活框处理、状态转换条件 |
| 人工审查 | 不可省略——AI 是助手，不是替代者 |

> "模型是思考的工具。AI 生成的是初稿，工程师的审查赋予它正确性。"
